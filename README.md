# SwiftDeploy

SwiftDeploy is a DevOps automation project that deploys a containerized Python API behind an Nginx reverse proxy using Docker Compose.

The project uses a manifest-driven approach. Instead of manually editing deployment files, the `swiftdeploy` CLI reads `manifest.yaml`, generates Docker Compose and Nginx configuration files, validates the environment, deploys the application, supports stable/canary promotion, allows controlled chaos testing, and provides a teardown command for cleanup.

---

## Project Overview

SwiftDeploy uses `manifest.yaml` as the source of truth for deployment configuration.

From the manifest, the CLI generates:

- `generated/docker-compose.yml`
- `generated/nginx.conf`

The deployment contains two main containers:

1. **Application container**: a Python Flask API served with Gunicorn.
2. **Nginx container**: a reverse proxy that exposes the application on port `8080`.

The application container is not directly exposed to the host machine. Only Nginx is exposed publicly, while the app runs privately inside the Docker network.

---

## Architecture

Client or curl request goes to Nginx on port `8080`. Nginx then forwards the request internally to the Flask API running on port `3000` inside the Docker network.

Flow:

Client → Nginx Reverse Proxy → Docker Network → Flask/Gunicorn API

## Folder Structure

The project is organized as follows:

- `app/`: Flask application source code
- `templates/`: template files for Docker Compose and Nginx
- `generated/`: deployment files generated by the CLI
- `Dockerfile`: builds the application image
- `manifest.yaml`: deployment configuration source of truth
- `swiftdeploy`: CLI automation script
- `README.md`: project documentation

## Manifest Configuration

The `manifest.yaml` file controls the deployment.

Important values include:

- `project.name`: project name
- `project.version`: project version
- `project.contact`: project contact email
- `services.name`: application container name
- `services.image`: Docker image used by the app
- `services.port`: internal app port
- `services.mode`: stable or canary
- `services.app_version`: application version
- `services.restart_policy`: container restart policy
- `nginx.image`: Nginx image
- `nginx.port`: public Nginx port
- `nginx.proxy_timeout`: proxy timeout value
- `network.name`: Docker network name
- `network.driver_type`: Docker network driver
- `logs.volume_name`: log volume name

The `services.mode` value controls whether the app runs in `stable` or `canary` mode.

## Application Endpoints

### Root Endpoint

`GET /`

This returns the current deployment mode, application version, and timestamp.

Example response includes:

- `message`
- `mode`
- `version`
- `timestamp`

---

### Health Check Endpoint

`GET /healthz`

This returns the health status of the app.

Example response includes:

- `status`
- `mode`
- `version`
- `uptime_seconds`

---

### Chaos Endpoint

`POST /chaos`

The chaos endpoint only works in canary mode. It supports controlled failure testing.

Supported modes:

- `slow`
- `error`
- `recover`

Example slow test:

`curl -i -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"slow","duration":2}'`

Recover from chaos mode:

`curl -i -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"recover"}'`

## CLI Commands

SwiftDeploy is controlled using the `swiftdeploy` CLI.

Available commands:

- `init`: generate Docker Compose and Nginx files
- `validate`: run deployment pre-flight checks
- `deploy`: deploy the stack
- `promote`: promote between stable and canary mode
- `teardown`: remove containers, network, and volumes

On Windows Git Bash, the CLI can be run with:

`python swiftdeploy <command>`

Example:

`python swiftdeploy validate`

---

## Build the Docker Image

Before deployment, build the application image:

`docker build -t swift-deploy-1-node:latest .`

Confirm the image exists:

`docker images | grep swift-deploy-1-node`

---

## Generate Deployment Files

Run:

`./swiftdeploy init`

This generates:

- `generated/docker-compose.yml`
- `generated/nginx.conf`

These files are created from templates using values from `manifest.yaml`.

---

## Validate the Deployment

Run:

`./swiftdeploy validate`

The validation command checks that:

- The manifest exists
- Required manifest keys are present
- Template files exist
- Generated files exist
- Docker Engine is reachable
- Docker Compose is available
- The application image exists locally
- The generated Docker Compose file is valid
- Nginx includes the `X-Deployed-By: swiftdeploy` header
- The app service is internal-only
- Container security hardening is enabled
- Port `8080` is available

This helps catch errors before deployment.

---

## Deploy the Stack

Run:

`./swiftdeploy deploy`

The deploy command:

1. Regenerates the deployment files.
2. Runs validation checks.
3. Starts the containers using Docker Compose.
4. Waits for `/healthz` to become healthy.
5. Prints the local deployment URL.

The application becomes available at:

`http://localhost:8080`

---

## Test the Stable Deployment

Run:

`curl -i http://localhost:8080/`

Expected signs of success:

- `HTTP/1.1 200 OK`
- `X-Deployed-By: swiftdeploy`
- Response body includes `"mode":"stable"`

Test the health endpoint:

`curl -i http://localhost:8080/healthz`

The response should include `"status":"ok"`.

---

## Promote to Canary

Run:

`./swiftdeploy promote canary`

Then test:

`curl -i http://localhost:8080/`

Expected signs of success:

- `X-Deployed-By: swiftdeploy`
- `X-Mode: canary`
- Response body includes `"mode":"canary"`

This confirms that the app is running in canary mode.

---

## Run Chaos Testing

Chaos testing is available only in canary mode.

Run a slow response test:

`curl -i -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"slow","duration":2}'`

Recover from chaos mode:

`curl -i -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"recover"}'`

---

## Promote Back to Stable

Run:

`./swiftdeploy promote stable`

Confirm stable mode:

`curl -i http://localhost:8080/`

The response body should include `"mode":"stable"`. The `X-Mode: canary` header should no longer appear.

---

## Teardown

To remove the deployment, run:

`./swiftdeploy teardown`

This removes the containers, Docker network, named volumes, and orphaned services.

---

## Security Notes

SwiftDeploy includes basic container security practices:

- The app runs as a non-root user inside the container.
- The app container does not publish a host port.
- Nginx is the only public entry point.
- Linux capabilities are dropped in Docker Compose.
- `no-new-privileges:true` is enabled.
- The app image uses a slim Python base image.
- Nginx returns structured JSON error responses for upstream failures.

---

## Troubleshooting

### Docker Engine is not reachable

Open Docker Desktop and wait until Docker Engine is running.

Then run:

`docker version`

A working setup should show both `Client` and `Server`.

### Port 8080 is already in use

Stop the existing stack:

`./swiftdeploy teardown`

Or change the Nginx port in `manifest.yaml`.

---

## Project Defence Summary

SwiftDeploy demonstrates a practical DevOps deployment workflow.

The project shows how to use a manifest as the source of truth, generate deployment files from templates, validate the environment before deployment, deploy containers with Docker Compose, route traffic through Nginx, keep the app service private inside a Docker network, promote between stable and canary modes, run controlled chaos testing, and tear down the environment cleanly.

The key design decision is that stable and canary promotion is controlled by configuration, not by rebuilding the Docker image. This makes the deployment process more repeatable, safer, and closer to real-world DevOps practices.
